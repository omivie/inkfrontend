# Dense-pack Rollout — May 2026

**Backend handoff date:** 2026-05-06
**Backend release:** pack-resolver fix that surfaces ~232 previously-filtered packs
**Compatibility:** **fully additive** — no API contract change, just more rows

Companion read: [`api-changes-may2026.md`](./api-changes-may2026.md) §5 (pack
rendering — more legitimate packs survive) and §7 (pack-guard rollout). The
rollout in this doc is the practical follow-up to those server-side changes.

---

## What changed on the backend

`detectStalePacks` (the orphan-deactivation sweep that runs daily and on every
import) was incorrectly filtering ~232 valid packs out of every product-list
endpoint. After the fix, those packs reappear on:

- `GET /api/shop`
- `GET /api/search/smart`
- `GET /api/search/by-printer`
- `GET /api/search/by-part`
- `GET /api/products/printer/:slug`
- `GET /api/printers/:slug/products`
- `GET /api/prerender/printer/:brandSlug/:printerSlug`
- `GET /api/prerender/category/:category`

No API contract change — same response shape, same fields. Just more rows.

The packs were always real products. Examples per `api-changes-may2026.md` §5:
HP 63 KCMY tri-color, Brother DR233CL CMY drum, Lexmark 73D0Q00 toner, plus
letter-encoded color in series (`G-BRO-LC536C-INK-CY`,
`G-CAN-CART329BK-TNR-BK`), generalised digit-suffix
(`G-FUJ-CT201370`-…-`CT201373` → `CT20137`), and legacy
`<base>-INK-KCMY-4PK` SKU shapes.

## Frontend impact — what was actioned

### 1. Cache busting

| Surface | Behaviour | Action |
| --- | --- | --- |
| Static HTML (`/shop`, `/html/*`) | Vercel default — `max-age=0, must-revalidate` | None. Re-fetched on every navigation; reads via API. |
| `/js/*`, `/css/*` | `max-age=0, must-revalidate` + `CDN-Cache-Control: no-cache` (`vercel.json`) | None. Frontend redeploy busts edge cache automatically. |
| API SWR in-memory cache | 60s TTL (`api.js` `SWR_TTL_MS`) | Self-heals within 1 minute of any change. |
| Drilldown chip cache (`shop-page.js`) | Bumped to `codes-v6` for May 2026 catalog overhaul | Already wired; key invalidates stale entries on first mount. |
| **Bot-prerender HTML** (middleware.js) | Was `s-maxage=3600, max-age=3600` → could stay stale up to 1h | **Added `stale-while-revalidate=86400`** so the next crawler hit past expiry serves stale immediately AND triggers a background refresh. Pinned by `tests/dense-pack-rollout-may2026.test.js` §3. |
| Sitemap, sub-sitemaps, product feeds (`/sitemap.xml`, `/feeds/*`) | Proxied directly to backend via `vercel.json` rewrites | None. Backend regenerates on its own deploy cadence; new packs appear automatically. |

### 2. Sitemap regeneration

Sitemaps are **owned by the backend**. The frontend rewrites:

```json
{ "source": "/sitemap.xml",            "destination": "https://ink-backend-zaeq.onrender.com/sitemap.xml" }
{ "source": "/sitemap-:path.xml",      "destination": "https://ink-backend-zaeq.onrender.com/sitemap-:path.xml" }
{ "source": "/feeds/google-shopping.xml",   "destination": "https://ink-backend-zaeq.onrender.com/feeds/google-shopping.xml" }
{ "source": "/feeds/facebook-catalog.tsv",  "destination": "https://ink-backend-zaeq.onrender.com/feeds/facebook-catalog.tsv" }
{ "source": "/feeds/google-promotions.xml", "destination": "https://ink-backend-zaeq.onrender.com/feeds/google-promotions.xml" }
```

After the backend regenerates its sitemap (catalog → SQL → XML pipeline), the
new pack URLs appear in `/sitemap.xml` automatically. No frontend action.
Pinned by `tests/dense-pack-rollout-may2026.test.js` §4.

### 3. Pagination / grid layouts

Audited every product-rendering surface:

| Surface | Cap | Layout | Notes |
| --- | --- | --- | --- |
| Drilldown per-code (`shop-page.js loadProducts`) | `limit: 200` | `.products-row` (`flex-wrap`) | Densest realistic per-code list is ~22 products (Brother LC536). 200 is generous. |
| Drilldown per-printer (`loadPrinterProducts`) | No client cap (backend default) | `.products-row` | Backend ships full compatible-products list. Brother MFC-J6945DW = 27 products. Flex-wrap handles any count. |
| PDP related products (`renderRelatedProducts`) | `limit: 200` | `.related-products__grid product-grid` | Re-uses `/api/shop` with the matching code chip. Same ceiling as drilldown. |
| Free-text smart search (`shop-page.js`) | `limit: 100` | `.products-row` | Search results page. |
| Recovery rails (search miss) | `limit: 6` per by-printer rail; `slice(0, 4)` for compat-printers | `.recovery-rail` | Slice-bounded — visual count fixed. |
| Featured products (homepage) | `limit: 8` | Hero grid | Slice-bounded. |
| Bought-together (PDP) | `slice(0, 4)` | `.bought-together__grid` | Slice-bounded. |
| Color-pack bundles (printer page) | Backend-shaped | `#color-packs-grid` | Already auto-generated by backend; just reflects the new pack inventory. |

`flex-wrap` handles any number of cards without overflow. No pagination on the
codes-level product list — by design, since a code/series rarely exceeds ~25
products even with the new packs included. Pinned by
`tests/dense-pack-rollout-may2026.test.js` §2.

### 4. Carousels / "related products" rails

Every carousel / rail consumes one of the 8 endpoints listed above and renders
in API order — no client-side `.sort()` applied to the response. The May 2026
sort hierarchy (`sortByCatalogOrder` / `sortByRelevance`) lives entirely on
the server, and the frontend's render-in-API-order rule is pinned by:

- `tests/api-changes-may2026.test.js` §1 (shop drilldown, PDP related, recovery)
- `tests/dense-pack-rollout-may2026.test.js` §1 (printer-products, recovery rails)

When the new packs surface, they slot into the right tier (after singles per
sort key `(accessoryTier, yieldTier, seriesBase, colorOrder, packRank, name)`).
No re-sort or de-duplication is needed — the `pack_type` ribbon
("Value Pack" / "Multipack") already differentiates pack rows visually
(`shop-page.js` `packTypeRibbon`).

## Slug / URL changes

Per the backend handoff: **none**. The rename-style repair was deliberately
held off pending frontend sign-off. All inbound links (`/products/<slug>/<sku>`,
`/p/<sku>`, brand → `/shop?brand=…`, printer → `/shop?printer_slug=…`)
continue to resolve. Genuine images, JSON-LD shapes, and the pack-guard runtime
contract are unchanged.

## Why this round needed no frontend logic change

The frontend was already correct for dense pack lists, because the May 2026
catalog overhaul (`api-changes-may2026.md`) had pre-loaded the architecture:

- **No client-side product `.sort()`** on any API response.
- **Generous per-code/per-search limits** (200, 100) above the densest
  realistic page.
- **Slice-bounded rails** (4, 6, 8) so visual layout is fixed regardless of
  how many rows the API returns.
- **`pack_type` ribbon and pack-aware card rendering** already in place.
- **Sitemap and feeds proxied to backend**, so the canonical URL list refreshes
  with the catalog on every backend deploy.

The two real deltas: (a) a `stale-while-revalidate` directive on the
bot-prerender Cache-Control so post-deploy crawler HTML refreshes within one
hit instead of waiting out s-maxage, and (b) this doc + the test file pinning
the invariants so the next backend rollout doesn't have to re-prove them.
